# Provenance in ReconGraph

ReconGraph requires mathematical certainty when matching financial records. A core part of that certainty is **Provenance** — the ability to trace any reconciliation decision backward to the exact inputs, engine rules, and domain heuristics that caused it.

## The Provenance Chain
The provenance of a decision flows logically backwards through the system architecture:

1. **Decision** (`ReconciliationDecision`): The final outcome (e.g. `AUTO_MATCH`).
2. **FusionResult**: The evaluated state of the graph.
3. **Propagation**: How evidence traversed the graph edges.
4. **EvidenceContribution**: The bipolar support/conflict generated by a specific provider.
5. **Projection**: The domain-specific mapping of observations.
6. **Interpretation**: The semantic meaning extracted from observations.
7. **Observation**: The clean, parsed artifacts.
8. **Parser**: The code that extracted the artifacts.
9. **Raw Record**: The original unstructured data (Purchase/GST).

## Provenance Enforcement

ReconGraph enforces provenance structurally. Every `EvidenceContribution` explicitly stores its originating `provider_name` and the specific `violations` it found. These properties are locked into an `identity_hash` within the `FusionNode`.

When building the graph, the `EvidenceGraph` nodes track exactly where they came from.

## The Provenance Contract
If an explanation artifact states that a Match failed because the Tax identities conflicted, the Provenance Contract dictates that there MUST exist a cryptographic trail back to a `TAX_NODE` with a `CONFLICT` contribution state, which MUST map to a specific `tax_identity_conflict` finding. 

This strict contract eliminates silent failures, undocumented behavior overrides, and heuristic hallucinations.
